# PublicVesting — Общедоступный вестинг с распределением комиссий на Bitcoin L1

Любой владелец OP_20 токенов может создать график вестинга для любого получателя за одну транзакцию. Доход от внесенных комиссий пропорционально распределяется между всеми держателями заблокированных токенов с использованием O(1) аккумулятора Synthetix — никаких циклов и никаких ограничений со стороны владельца контракта.

**Приложение:** https://public-vesting.vercel.app/

---

## Особенности (Features)

- **`depositAndVest(amount, beneficiary, cliff, duration)`** — полностью публичный метод, не требует прав владельца
- **`depositRevenue(amount)`** — любой адрес может направить комиссии протокола для пропорционального распределения
- **`claimRevenue()`** — получатели могут забрать свою накопленную долю дохода
- **`release()`** — получатель высвобождает все заработанные (vested) токены после клиффа (cliff)
- O(1) аккумулятор наград на токен по модели Synthetix — никаких итераций по спискам держателей
- Защита от повторного входа (reentrancy guard) через `StoredBoolean` (постоянное хранилище, переживает кросс-контрактные вызовы)
- Строгое соблюдение шаблона CEI (Checks-Effects-Interactions) в каждом методе, изменяющем состояние
- Единый токен OP_20 используется как для вестинга, так и для распределения дохода

## Методы смарт-контракта

| Метод | Доступ | Описание |
|---|---|---|
| `initialize(revenueToken)` | Владелец, один раз | Устанавливает адрес OP_20 токена |
| `depositAndVest(amount, beneficiary, cliff, duration)` | Публичный | Блокирует токены, создает график вестинга |
| `release()` | Получатель | Высвобождает доступные токены из вестинга |
| `depositRevenue(amount)` | Публичный | Вносит доход для пропорционального распределения |
| `claimRevenue()` | Получатель | Забирает накопленную долю дохода |
| `getVesting(address)` | View | Возвращает полный график вестинга для адреса |
| `getPendingRelease(address)` | View | Возвращает сумму, доступную для высвобождения в данный момент |
| `getClaimableRevenue(address)` | View | Возвращает сумму дохода, доступную для снятия адресом |
| `totalLocked()` | View | Общее количество заблокированных токенов по всем графикам |
| `totalRevenueDeposited()` | View | Совокупный доход, внесенный за все время |
| `revenueToken()` | View | Адрес токена |
| `owner()` | View | Владелец контракта |

## Сборка и развертывание (Build & Deploy)

### Предварительные требования

```bash
npm install
```

### Компиляция

```bash
npm run build          # отладочная сборка  →  build/feevest.debug.wasm
npm run build:release  # релизная сборка → build/feevest.release.wasm
```

### Развертывание в тестовой сети OPNet

```bash
# Развертывание контракта (без данных конструктора — используется паттерн initialize())
opnet deploy \
  --network testnet \
  --wasm build/feevest.release.wasm \
  --gasSatFee 10000

# После развертывания вызовите initialize один раз от имени владельца:
# feeVest.initialize(revenueTokenAddress)
```

> **Примечание:** Тестовая сеть OPNet передает 0 байт в `onDeploy()`, поэтому настройка токена выполняется через единоразовый вызов `initialize()` после развертывания. Используйте вкладку Admin в панели управления (dashboard).

### Минимальный газ

Используйте `gasSatFee: 10_000n` (или выше для больших WASM). Значения ниже этого могут привести к скрытой отмене транзакции (revert silently).

## Архитектура

```
depositAndVest()
  ├─ CHECKS:  проверка входных данных, проверка отсутствия существующего графика
  ├─ EFFECTS: снепшот updateReward, запись графика, увеличение totalLocked
  └─ INTERACT: transferFrom(отправитель → хранилище)

depositRevenue()
  ├─ CHECKS:  amount > 0, totalLocked > 0
  ├─ EFFECTS: rewardPerToken += (amount × 1e18) / totalLocked
  └─ INTERACT: transferFrom(отправитель → хранилище)

release()
  ├─ CHECKS:  есть график, releasable > 0
  ├─ EFFECTS: updateReward, отметка о высвобождении, уменьшение totalLocked, перепривязка долга
  └─ INTERACT: transfer(хранилище → получатель)

claimRevenue()
  ├─ CHECKS:  есть график
  ├─ EFFECTS: updateReward, обнуление pendingRewards
  └─ INTERACT: transfer(хранилище → получатель)
```

## Безопасность

- Аудит: 0 критических уязвимостей — Bob MCP + ручная проверка
- Защита от повторного входа `StoredBoolean` (это не поле класса — состояние сохраняется между фреймами вызовов)
- Отсутствие `tx.origin` — используется только `Blockchain.tx.sender`
- Вся арифметика u256 осуществляется через `SafeMath`
- Нет циклов по неограниченным данным — O(1) везде
- Один активный график на одного получателя (защита от гриффинга/griefing)
